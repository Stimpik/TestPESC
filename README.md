# Auth Service с JWT и Redis (чёрный/белый списки)

##  Описание проекта

REST API сервис аутентификации и авторизации на **FastAPI** с использованием **JWT** (access + refresh) и **Redis** для управления:

- **Чёрный список (blacklist)** – отозванные access-токены (логаут, смена пароля).
- **Белый список (whitelist)** – активные refresh-токены с ротацией.

Реализована ролевая модель (`user`, `super_user`) и два типа контента: общий для всех аутентифицированных и доступный только администраторам.

##  API-endpoints

| Метод | Эндпоинт                 | Назначение                                   |
|-------|--------------------------|----------------------------------------------|
| POST  | `/users/`                | Регистрация пользователя                     |
| POST  | `/users/token`           | Логин (выдача access + refresh)              |
| POST  | `/users/refresh-token`   | Обновление пары токенов (ротация)            |
| POST  | `/users/logout`          | Логаут (добавление access в blacklist)       |
| GET   | `/content/common`        | Общий контент (доступен всем аутентифицированным) |
| GET   | `/content/super`         | Супер-контент (только `super_user`)          |


## Система ролей

- **`user`** – обычный пользователь, имеет доступ только к `/content/common`.
- **`super_user`** – супер-пользователь, дополнительно имеет доступ к `/content/super` и `/users/users`.
- Неаутентифицированным пользователям контент не доступен. 

При регистрации роль всегда назначается `user`. Для создания `super_user` необходимо предусмотрен отдельный эндпоинт (в примере не реализован для простоты).

## Безопасность и управление токенами

### Чёрный список (blacklist)

- При логауте `jti` access-токена сохраняется в Redis с TTL, равным оставшемуся времени жизни токена.
- При каждом запросе к защищённому ресурсу проверяется, не находится ли `jti` в blacklist.

### Белый список (whitelist) refresh-токенов

- При логине refresh-токен сохраняется в Redis с ключом `refresh:{jti}` и TTL, соответствующим сроку жизни токена.
- При обновлении токенов старый refresh удаляется, новый сохраняется – происходит ротация.
- Повторное использование старого refresh-токена приводит к ошибке `Refresh token revoked`.
